## Context

### 当前状态
- SessionManager.add_message() 存消息到 sessions.db
- Memory 存在 memory.db（通过 memory_tools.remember()）
- 用户需要手动调用 remember tool

### 问题
- 很多有价值的信息隐式存在于对话中，用户不会主动说"记住"
- 需要自动扫描对话，识别值得记录的信息

### 约束
- 不阻塞主流程（用户响应时间不能增加）
- 成本可控（不能每条消息都调用 LLM）
- 复用现有 memory_tools.remember()

## Goals / Non-Goals

**Goals:**
- 每 5 个 user 消息触发一次检查
- 检查最近 20 条消息（包含完整对话）
- 智能判断值得记录的内容
- 后台异步执行，用户无感知

**Non-Goals:**
- 不实现跨 session 的记忆关联
- 不实现记忆去重（后续迭代）
- 不实现用户配置开关（后续迭代）

## Decisions

### D1: 触发时机 - 消息计数

**决策**: 每 5 个 user 消息触发

**理由**:
- 3 个太频繁，成本高
- 10 个太稀疏，容易错过时机
- 5 个适中，覆盖完整对话片段

**备选方案**:
- 方案 A（采纳）: 每 5 个 user 消息触发
- 方案 B（拒绝）: 每 10 条消息触发（包含 assistant/tool，太稀疏）
- 方案 C（拒绝）: 定时扫描（效率低，可能错过时机）

### D2: 检查范围 - 最近 20 条消息

**决策**: 获取最近 20 条消息（包含所有角色）

**理由**:
- 5 个 user 消息约对应 5 个 Turn
- 每个 Turn 约 4-5 条消息（user + assistant + tool）
- 20 条保证覆盖完整对话上下文

**备选方案**:
- 方案 A（采纳）: 最近 20 条（包含所有角色）
- 方案 B（拒绝）: 只看 user 消息（缺少上下文）

### D3: 计数器清零方式

**决策**: 触发后清零

**理由**:
- 逻辑清晰（每次凑够 5 个就触发）
- 不依赖大数判断（`count % 5 == 0`）

**备选方案**:
- 方案 A（采纳）: 清零方式
- 方案 B（拒绝）: 不清零，用 `count % 5 == 0`（逻辑稍复杂）

### D4: 智能判断方式

**决策**: 关键词检测 + LLM 分析

**理由**:
- 关键词检测作为第一层筛选（成本低）
- LLM 分析作为第二层深度判断（成本可控）

**关键词列表**:
- 中文: "记住", "请记住", "不要", "必须", "禁止", "我喜欢", "我习惯"
- 英文: "remember", "always", "never", "don't", "I prefer", "I like"

**备选方案**:
- 方案 A（采纳）: 关键词 + LLM 混合
- 方案 B（拒绝）: 只用关键词（可能漏掉隐式偏好）
- 方案 C（拒绝）: 只用 LLM（成本太高）

### D5: 实现位置

**决策**: 在 SessionManager.add_message() 中触发

**理由**:
- add_message() 是存消息的唯一入口
- 可以直接获取 session 和消息数据
- 可以启动后台 asyncio.create_task()

**备选方案**:
- 方案 A（采纳）: SessionManager.add_message() 内部
- 方案 B（拒绝）: app.py 中单独调用（需要额外同步）
- 方案 C（拒绝）: Hook 系统（shell 进程开销，不适合访问 Python 对象）

## Risks / Trade-offs

### [R1] LLM 成本增加
- **风险**: 每 5 个 user 消息触发一次 LLM 调用
- **缓解**: 关键词先筛选，减少 LLM 调用次数；后续可增加用户配置开关

### [R2] 记忆重复
- **风险**: 用户多次表达相似偏好，可能重复记录
- **缓解**: 后续迭代增加去重机制

### [R3] 计数器线程安全
- **风险**: 多线程并发 add_message() 可能导致计数器不准
- **缓解**: SQLite connection 是线程安全的；或使用 asyncio.Lock

### [R4] 检查不完整
- **风险**: 触发时对话可能还在进行（第 5 个 user 消息后还有后续）
- **缓解**: 检查最近 20 条，保证覆盖完整 Turn
